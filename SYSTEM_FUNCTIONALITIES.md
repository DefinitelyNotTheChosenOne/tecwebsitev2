# 🌌 TutorMatch Platform System Capabilities
*Confidential Deployment Intelligence & Feature Manifest*

This document outlines the complete scope of features, user flows, and backend logic currently active in the application.

---

## 1. Authentication & Role Protocols
*The foundation of the platform relies on strict role-based access control.*

* **Supabase Authentication**: Secure login and session management.
* **Three-Tier User Hierarchy**:
  * **Buyer (Student)**: Can request help, join sessions, browse subjects, and report infractions.
  * **Seller (Tutor/Specialist)**: Can build professional portfolios, negotiate classes, and charge for time.
  * **Systems Admin**: High Command override. Moderates disputes, manages global architecture.
* **Role Routing**: The application forces users to their appropriate dashboard instances depending on their role upon login.

---

## 2. Advanced Subject & Taxonomy System
*A smart-routing mechanism that ensures students only see available talent.*

* **Admin Subject Architect**: Admins can deploy new core subjects via `/admin/subjects`.
* **Dynamic Profile Linking**: When Sellers edit their profiles, they select their subjects from the Admin's master list.
* **Intelligent Auto-Hiding**: The public Markets (Homepage and `/subjects`) query a custom SQL RPC (`get_active_subjects()`). A subject will **not** appear to the public unless there is at least one active tutor hosting it.

---

## 3. Specialist Profiles & Credentialing
*Tutors can sculpt their public-facing appearance to optimize conversion.*

* **Public Portfolios** (`/tutors/[id]`): Shareable pages showcasing a tutor's bio, specialized skills, hourly rate, and encrypted credentials.
* **Dossier Vault System**: Within `/dashboard/profile`, sellers can upload standard verification documents (Resumes/Transcripts). Payload limits (max 25MB).
* **Avatar & Intel**: Fast updates for names, URLs, and bios that immediately reflect on the marketplace.

---

## 4. The Request Board (Help Wanted)
*An inverse-marketplace where students post needs and tutors apply.*

* **Post Requests**: Students can declare an open request (e.g. "I need help with Physics on Friday").
* **Smart Matching**: The Seller's Dashboard actively scans the request board. If a student posts a request matching a Seller's chosen `skills`, it will appear in the Seller's "Recommended Missions".

---

## 5. Next-Gen Communication Terminal (The Session Page)
*A multi-stage negotiation and execution environment. Built dynamically in `/sessions`.*

* **Updates Tab**: Tracks the connection state (Pending, Accepted, Declined) and outlines upcoming scheduled class missions.
* **Discussion Tab**: Real-time Supabase WebSockets chat for open negotiation (price, availability).
    * **Reporting Mechanism**: Built-in "Report Issue" toggle allowing students to select specific offending chat bubbles and escalate them (Sexually Explicit, Scam, Off-platform transactions) straight to the Admin.
* **Live Class Tab (Time-Locked)**: A strict, distraction-free execution environment. This terminal remains completely locked until the exact start-time of the scheduled class. Features its own standalone live-chat database.
* **History Tab**: An ongoing archive of previously completed classes.

---

## 6. Admin High Command Dashboard
*Total oversight for platform safety and stability (`/admin/dashboard`).*

* **Active Disputes Engine**: A UI to process class conflicts, track order IDs, read the reasons for disputes, and either "Release Funds" or "Refund Buyer".
* **Flagged Content Hub**: Receives the specific payloads generated by students who use the "Report Issue" button. 
* **Global Intelligence Metrics**: Live UI for transactional velocity, daily active users, and system capacities.
* **Direct Access Management**: Overriding user strikes or directly intervening in live student-tutor tunnels.

---

## 7. Performance & Aesthetic Routing
*The framework optimization applied to give it a "10x" feel.*

* **Surgical Navigation**: Modified Framer Motion `popLayout` transitions to prevent blank white screens during high-speed routing.
* **Cache Revalidation**: Clean cache intercepts on authentication (switching `router.push` to `window.location.href`) guaranteeing seamless dashboard injection without hanging states.
* **Dynamic Backgrounds**: Glassmorphism, tailored drop-shadows, pulsing connection indicators, and raw typographic hierarchies designed for a premium, specialized aesthetic.
